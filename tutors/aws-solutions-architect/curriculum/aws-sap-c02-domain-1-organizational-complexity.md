# Domain 1: Design Solutions for Organizational Complexity

Weight: 26%

## Unit goal

Build the ability to design AWS architectures that work across multiple teams, accounts, networks, compliance boundaries, and operating models.

## Learning objectives

| Objective ID | Learning objective | Focus areas | Recommended references |
| --- | --- | --- | --- |
| SAP-D1-O1 | Design a multi-account operating model that separates workloads, environments, and shared services without creating unnecessary management overhead. | Account boundaries, shared services, delegated operations, governance posture | SAP-C02 exam guide, AWS Architecture Center, AWS Well-Architected Framework |
| SAP-D1-O2 | Choose governance and guardrail patterns that balance central control with team autonomy. | Policies, preventive vs detective controls, delegated administration, auditability | SAP-C02 exam guide, AWS Well-Architected Framework, Solutions Architect ramp-up guide |
| SAP-D1-O3 | Design network connectivity patterns for large organizations with hybrid, multi-account, and multi-Region requirements. | Transit patterns, segmentation, DNS strategy, private connectivity, edge considerations | SAP-C02 in-scope services page, AWS Architecture Center, AWS Well-Architected Framework |
| SAP-D1-O4 | Design identity, security, and observability approaches that scale across business units and compliance domains. | Federation, centralized logging, key management, security services, evidence collection | SAP-C02 exam guide, AWS Well-Architected Framework, AWS Prescriptive Guidance |
| SAP-D1-O5 | Evaluate enterprise tradeoffs created by regional expansion, acquisitions, divestitures, and varying regulatory constraints. | Data residency, operating model change, service control boundaries, resilience vs complexity | SAP-C02 exam guide, AWS Architecture Center, AWS Prescriptive Guidance |

## Typical mastery signals

- Can explain why an organization would choose one account and network boundary model over another.
- Can compare governance options in terms of control, speed, auditability, and blast radius.
- Can defend architecture decisions in scenarios with conflicting business-unit requirements.
